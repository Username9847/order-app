# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 프런트엔드 UI 요구사항

### 4.1 주문하기 화면

#### 4.1.1 화면 목적
- 목적: 사용자가 커피 메뉴를 선택하고 옵션을 설정한 뒤 장바구니에 담아, 한 번에 주문을 요청할 수 있는 화면
- 대상 사용자: 일반 사용자(매장 직원/고객 공용 사용 가정)
- 성공 기준
  - 사용자는 3클릭 이내에 기본 커피 1잔을 장바구니에 담을 수 있어야 함
  - 장바구니에 담긴 메뉴와 총 금액을 한눈에 확인할 수 있어야 함

#### 4.1.2 주요 사용자 시나리오
- 시나리오 A – 기본 메뉴 주문
  - 사용자는 앱 상단에서 `주문하기` 탭이 선택된 상태로 진입
  - 커피 카드에서 원하는 메뉴를 확인하고 `담기` 버튼을 눌러 장바구니에 추가
  - 장바구니에서 선택한 메뉴와 금액을 확인하고 하단 `주문하기` 버튼을 눌러 주문 생성
- 시나리오 B – 옵션이 포함된 메뉴 주문
  - 사용자는 특정 메뉴 카드에서 옵션(예: `샷 추가(+500원)`, `시럽 추가(+0원)`) 체크박스를 선택
  - `담기` 버튼을 눌러 옵션이 포함된 메뉴를 장바구니에 추가
  - 장바구니에서 옵션명이 메뉴명 옆에 표시된 것을 확인
- 시나리오 C – 동일 메뉴 여러 잔 주문
  - 사용자는 같은 메뉴/옵션 조합에 대해 `담기` 버튼을 여러 번 눌러 수량을 증가
  - 장바구니에는 동일 메뉴/옵션 조합이 하나의 라인으로 표시되고, 수량(`X n`)과 합계 금액이 갱신

#### 4.1.3 정보 구조 & 레이아웃
- 상단 네비게이션 바
  - 좌측: 앱 이름 표시(예: `COZY`)
  - 우측: 탭 버튼 2개
    - `주문하기` 탭: 현재 화면, 선택 상태 스타일(배경/테두리 강조)
    - `관리자` 탭: 비선택 상태, 클릭 시 관리자 화면으로 라우팅
- 메뉴 리스트 영역
  - 카드 형태의 커피 메뉴 리스트(반응형일 경우 여러 줄로 배치)
  - 각 카드 구성
    - 썸네일 영역(이미지 또는 플레이스홀더)
    - 메뉴명(예: `아메리카노(ICE)`, `아메리카노(HOT)`, `카페라떼`)
    - 기본 가격(예: `4,000원`, `5,000원`)
    - 짧은 설명(1줄, 예: `간단한 설명...`)
    - 옵션 체크박스 리스트(예: `샷 추가(+500원)`, `시럽 추가(+0원)` 등)
    - 카드 하단 `담기` 버튼
- 장바구니 영역(화면 하단)
  - 제목 `장바구니`
  - 라인 아이템 리스트
    - `메뉴명 (선택 옵션 요약)  X 수량` + 라인 소계 금액
  - `총 금액`과 전체 합계 금액(굵게 표시)
  - 주요 액션 버튼 `주문하기`

#### 4.1.4 컴포넌트별 상세 요구사항
- 상단 네비게이션
  - `COZY` 로고/텍스트 클릭 시: 주문하기 화면으로 이동
  - `주문하기` 버튼: 현재 페이지일 경우 클릭해도 동작 없음, 활성 스타일 표시
  - `관리자` 버튼: 클릭 시 관리자 화면 라우팅(예: `/admin`), 키보드 포커스/Enter로도 동작
- 메뉴 카드
  - 썸네일: 실제 이미지 또는 플레이스홀더, 비율 유지 및 반응형 축소
  - 메뉴명: 1~2줄 허용, 초과 시 말줄임표 처리
  - 가격: 원 단위 콤마 표기(예: `4,000원`)
  - 옵션 체크박스
    - 각 옵션은 독립적으로 선택 가능
    - 옵션 선택 시 라인 아이템 가격 계산에 반영됨
  - `담기` 버튼 동작
    - 클릭/Enter 시 현재 카드의 메뉴 + 선택된 옵션들을 장바구니에 추가
    - 성공 시: 사용자 피드백(토스트 등) 제공, 카드 옵션 체크 상태는 초기화(기본: 모두 해제)
    - 실패 시(재고 부족 등 서버 응답 기준): 실패 메시지 표시
- 장바구니 패널
  - 항목 추가/병합 규칙
    - 같은 메뉴 + 동일 옵션 조합: 기존 라인 수량 1 증가(라인 소계 금액 갱신)
    - 같은 메뉴 + 다른 옵션 조합: 별도 라인으로 추가
  - 총 금액 계산: 모든 라인 소계를 합산, 0원일 경우 `총 금액 0원` 표시
  - 빈 상태: `장바구니가 비어 있습니다.` 문구 표시, `주문하기` 버튼 비활성화
  - 편집 기능(1차 범위)
    - 수량 증가는 동일 아이템을 다시 `담기` 하는 방식으로만 제공
    - 삭제/감소 기능은 2차 확장 항목으로 둠(라인 삭제, `+`/`-` 수량 조절)
- 하단 `주문하기` 버튼
  - 활성 조건: 장바구니에 1개 이상 항목이 있을 때만 활성화
  - 클릭 시: 주문 생성 API 호출
    - 성공: 장바구니 초기화, 성공 메시지 표시(예: “주문이 접수되었습니다.”)
    - 실패: 실패 사유 기반 에러 메시지 표시
  - 로딩 상태: 주문 요청 중 버튼 비활성화 및 로딩 표시(예: `주문 처리중...`)

#### 4.1.5 프런트엔드 상태 및 데이터 모델(UI 관점)
- 메뉴 데이터(예시)
  - `id`, `name`, `temperature`, `basePrice`, `description`, `imageUrl`, `options[]`
  - `options[]`: `{ id, label, extraPrice }`
- 장바구니 아이템(예시)
  - `menuId`, `menuName`, `options[]`(선택 옵션), `unitPrice`, `quantity`, `lineTotal`
- 페이지 상태
  - `menus`: 메뉴 목록(fetch)
  - `cartItems`: 장바구니 상태(주문 완료 시 초기화)
  - `isSubmittingOrder`: 주문 요청 로딩 플래그
  - `errorMessage` 또는 토스트 상태

#### 4.1.6 예외/에러 처리
- 네트워크 오류
  - 메뉴 로딩 실패: 에러 메시지 + 재시도 버튼 표시
  - 주문 생성 실패: 토스트/모달로 실패 안내
- 유효성 검증
  - 장바구니가 비어 있으면 `주문하기` 버튼 비활성화(접근성 도구 입력에도 방어)
- 서버 에러(재고 부족/가격 불일치 등)
  - 서버 메시지가 있으면 우선 노출, 없으면 공통 메시지 노출

#### 4.1.7 반응형 & UI 가이드
- 브라우저 타겟: 데스크톱 웹(최소 1280px 폭 기준) 우선
- 반응형 규칙
  - 넓은 화면: 메뉴 카드 3개 이상 가로 배치
  - 좁은 화면: 2개/1개로 재배치하며 행 수 증가
  - 장바구니 패널: 하단 전체 폭 유지
- 스타일 가이드(고수준)
  - 카드/패널/버튼의 테두리/라운드/그림자 스타일 일관 유지
  - 주요 액션(`주문하기`) 버튼을 시각적으로 강조
  - 가격/총액은 굵게 표시

#### 4.1.8 추적 지표(선택)
- 기본 이벤트
  - 메뉴별 `담기` 클릭 횟수
  - `주문하기` 클릭 횟수 및 성공/실패 건수
- 파생 지표
  - 주문당 평균 메뉴 수량
  - 옵션 선택 비율(예: 샷 추가 선택 비율)

## 5. 백엔드 요구사항 (API & 데이터 모델)

### 5.1 데이터 모델

- **Menus**
  - `id`: PK, UUID 또는 숫자
  - `name`: 커피 이름 (예: 아메리카노(ICE))
  - `description`: 설명
  - `price`: 기본 가격(정수, 원 단위)
  - `image_url`: 상품 이미지 경로/URL
  - `stock_quantity`: 재고 수량(정수, 0 이상)
  - `created_at`, `updated_at`

- **Options**
  - `id`: PK
  - `menu_id`: FK → `Menus.id`
  - `name`: 옵션 이름 (예: 샷 추가)
  - `extra_price`: 옵션 추가 금액(정수, 원 단위)
  - `created_at`, `updated_at`

- **Orders**
  - `id`: PK
  - `created_at`: 주문 생성 시각
  - `status`: 주문 상태(`RECEIVED`, `IN_PROGRESS`, `COMPLETED`)
  - `total_amount`: 전체 주문 금액(정수, 원 단위)

- **OrderItems**
  - `id`: PK
  - `order_id`: FK → `Orders.id`
  - `menu_id`: FK → `Menus.id`
  - `menu_name_snapshot`: 주문 시점의 메뉴명 스냅샷
  - `unit_price`: 주문 시점의 단가(옵션 포함)
  - `quantity`: 수량
  - `line_total`: 합계 금액

- **OrderItemOptions**
  - `id`: PK
  - `order_item_id`: FK → `OrderItems.id`
  - `option_id`: FK → `Options.id`
  - `option_name_snapshot`: 주문 시점의 옵션 이름
  - `extra_price`: 주문 시점의 옵션 추가 금액

### 5.2 사용자 흐름과 데이터 스키마 연동

1. **메뉴 조회**
   - 클라이언트에서 `GET /api/menus` 호출.
   - 응답에는 `Menus` + 연결된 `Options` 정보 포함.
   - 응답 데이터 중 `stock_quantity`는 관리자 화면의 재고 영역에서 사용.

2. **장바구니 구성**
   - 사용자가 앱 화면에서 커피/옵션을 선택하면, 프런트엔드에서 장바구니 상태만 갱신(서버 호출 없음).
   - 장바구니 아이템 구조는 `menuId`, `menuName`, `optionIds[]`, `unitPrice`, `quantity`, `lineTotal` 등으로 구성.

3. **주문 생성**
   - 사용자가 `주문하기` 버튼 클릭 → `POST /api/orders` 호출.
   - 요청 바디에는 주문 시간은 서버에서 기록하고, **주문 내용(메뉴, 수량, 옵션, 금액)** 을 전송.
   - 서버는 다음을 수행:
     - 요청 검증 및 재고 확인 (`Menus.stock_quantity`)
     - `Orders`, `OrderItems`, `OrderItemOptions` 에 레코드 생성
     - 주문 기본 상태를 `RECEIVED` 로 저장
     - 사용한 수량만큼 `Menus.stock_quantity` 감소

4. **주문 현황 표시**
   - 관리자 화면에서 `GET /api/orders?status=RECEIVED,IN_PROGRESS` 를 호출.
   - 각 주문에 대해:
     - 주문 일시(`created_at`)
     - 주문 내용 요약(메뉴, 수량, 옵션)
     - 금액(`total_amount`)
     - 상태(`RECEIVED` / `IN_PROGRESS` / `COMPLETED`)
   - 프런트엔드는 이를 `주문 현황` 리스트로 렌더링하고, 상태별 카운트는 대시보드에서 사용.

5. **주문 상태 변경**
   - 관리자가 특정 주문의 상태를 변경할 때:
     - `PATCH /api/orders/:orderId/status`
     - 흐름: `RECEIVED` → `IN_PROGRESS` → `COMPLETED`
   - 서버는 상태 전이 유효성 검증 후 `Orders.status` 를 업데이트.

### 5.3 API 설계

#### 5.3.1 메뉴 조회 API

- **GET `/api/menus`**
  - **설명**: `주문하기` 화면 진입 시 커피 메뉴 목록을 불러오기.
  - **쿼리 파라미터**: 선택적으로 `includeOptions=true` 로 옵션 포함 여부 제어.
  - **응답 예시**:
    ```json
    [
      {
        "id": "americano-ice",
        "name": "아메리카노(ICE)",
        "description": "간단한 설명...",
        "price": 4000,
        "imageUrl": "/images/americano-ice.png",
        "stockQuantity": 8,
        "options": [
          { "id": "shot", "name": "샷 추가", "extraPrice": 500 },
          { "id": "syrup", "name": "시럽 추가", "extraPrice": 0 }
        ]
      }
    ]
    ```

#### 5.3.2 주문 생성 API

- **POST `/api/orders`**
  - **설명**: 사용자가 장바구니에서 `주문하기` 를 눌렀을 때 주문 생성.
  - **요청 바디**:
    ```json
    {
      "items": [
        {
          "menuId": "americano-ice",
          "quantity": 2,
          "optionIds": ["shot"]
        },
        {
          "menuId": "latte",
          "quantity": 1,
          "optionIds": []
        }
      ]
    }
    ```
  - **서버 처리**:
    - 재고 확인 (각 `menuId` 에 대해 `stock_quantity >= quantity` 확인)
    - 트랜잭션 내에서:
      - `Orders` 생성 (`status = RECEIVED`)
      - 각 장바구니 항목에 대해 `OrderItems`, `OrderItemOptions` 생성
      - `Menus.stock_quantity` 감소
  - **응답 예시**:
    ```json
    {
      "orderId": "ord-20260309-0001",
      "status": "RECEIVED",
      "createdAt": "2026-03-09T01:21:00.000Z",
      "totalAmount": 12500
    }
    ```

#### 5.3.3 주문 상세 조회 API

- **GET `/api/orders/:orderId`**
  - **설명**: 특정 주문 상세 정보 조회(관리자/추후 고객용).
  - **응답 예시**:
    ```json
    {
      "id": "ord-20260309-0001",
      "createdAt": "2026-03-09T01:21:00.000Z",
      "status": "RECEIVED",
      "totalAmount": 12500,
      "items": [
        {
          "menuName": "아메리카노(ICE)",
          "quantity": 1,
          "unitPrice": 4500,
          "lineTotal": 4500,
          "options": ["샷 추가"]
        }
      ]
    }
    ```

#### 5.3.4 주문 목록/현황 API

- **GET `/api/orders`**
  - **설명**: 관리자 화면 `주문 현황`에 사용할 주문 목록 조회.
  - **쿼리 파라미터**:
    - `status`: `RECEIVED`, `IN_PROGRESS`, `COMPLETED` 중 하나 또는 콤마 구분 여러 개.
  - **응답**: 주문 목록(최신순).

#### 5.3.5 주문 상태 변경 API

- **PATCH `/api/orders/:orderId/status`**
  - **설명**: 주문 상태를 `RECEIVED → IN_PROGRESS → COMPLETED` 순으로 변경.
  - **요청 바디**:
    ```json
    { "status": "IN_PROGRESS" }
    ```
  - **서버 처리**:
    - 현재 상태 확인 후 유효한 전이인지 검증.
    - `Orders.status` 업데이트.

#### 5.3.6 재고 수정 API

- **PATCH `/api/menus/:menuId/stock`**
  - **설명**: 관리자 화면에서 `+/-` 클릭 시 재고 조정.
  - **요청 바디**:
    ```json
    { "delta": 1 }
    ```
  - **서버 처리**:
    - 현재 재고에 `delta` 를 더한 값이 0 미만이면 0으로 고정.
    - `stock_quantity` 업데이트 후 최신 값 반환.
