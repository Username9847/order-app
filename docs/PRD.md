# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 프런트엔드 UI 요구사항

### 4.1 주문하기 화면

#### 4.1.1 화면 목적
- 목적: 사용자가 커피 메뉴를 선택하고 옵션을 설정한 뒤 장바구니에 담아, 한 번에 주문을 요청할 수 있는 화면
- 대상 사용자: 일반 사용자(매장 직원/고객 공용 사용 가정)
- 성공 기준
  - 사용자는 3클릭 이내에 기본 커피 1잔을 장바구니에 담을 수 있어야 함
  - 장바구니에 담긴 메뉴와 총 금액을 한눈에 확인할 수 있어야 함

#### 4.1.2 주요 사용자 시나리오
- 시나리오 A – 기본 메뉴 주문
  - 사용자는 앱 상단에서 `주문하기` 탭이 선택된 상태로 진입
  - 커피 카드에서 원하는 메뉴를 확인하고 `담기` 버튼을 눌러 장바구니에 추가
  - 장바구니에서 선택한 메뉴와 금액을 확인하고 하단 `주문하기` 버튼을 눌러 주문 생성
- 시나리오 B – 옵션이 포함된 메뉴 주문
  - 사용자는 특정 메뉴 카드에서 옵션(예: `샷 추가(+500원)`, `시럽 추가(+0원)`) 체크박스를 선택
  - `담기` 버튼을 눌러 옵션이 포함된 메뉴를 장바구니에 추가
  - 장바구니에서 옵션명이 메뉴명 옆에 표시된 것을 확인
- 시나리오 C – 동일 메뉴 여러 잔 주문
  - 사용자는 같은 메뉴/옵션 조합에 대해 `담기` 버튼을 여러 번 눌러 수량을 증가
  - 장바구니에는 동일 메뉴/옵션 조합이 하나의 라인으로 표시되고, 수량(`X n`)과 합계 금액이 갱신

#### 4.1.3 정보 구조 & 레이아웃
- 상단 네비게이션 바
  - 좌측: 앱 이름 표시(예: `COZY`)
  - 우측: 탭 버튼 2개
    - `주문하기` 탭: 현재 화면, 선택 상태 스타일(배경/테두리 강조)
    - `관리자` 탭: 비선택 상태, 클릭 시 관리자 화면으로 라우팅
- 메뉴 리스트 영역
  - 카드 형태의 커피 메뉴 리스트(반응형일 경우 여러 줄로 배치)
  - 각 카드 구성
    - 썸네일 영역(이미지 또는 플레이스홀더)
    - 메뉴명(예: `아메리카노(ICE)`, `아메리카노(HOT)`, `카페라떼`)
    - 기본 가격(예: `4,000원`, `5,000원`)
    - 짧은 설명(1줄, 예: `간단한 설명...`)
    - 옵션 체크박스 리스트(예: `샷 추가(+500원)`, `시럽 추가(+0원)` 등)
    - 카드 하단 `담기` 버튼
- 장바구니 영역(화면 하단)
  - 제목 `장바구니`
  - 라인 아이템 리스트
    - `메뉴명 (선택 옵션 요약)  X 수량` + 라인 소계 금액
  - `총 금액`과 전체 합계 금액(굵게 표시)
  - 주요 액션 버튼 `주문하기`

#### 4.1.4 컴포넌트별 상세 요구사항
- 상단 네비게이션
  - `COZY` 로고/텍스트 클릭 시: 주문하기 화면으로 이동
  - `주문하기` 버튼: 현재 페이지일 경우 클릭해도 동작 없음, 활성 스타일 표시
  - `관리자` 버튼: 클릭 시 관리자 화면 라우팅(예: `/admin`), 키보드 포커스/Enter로도 동작
- 메뉴 카드
  - 썸네일: 실제 이미지 또는 플레이스홀더, 비율 유지 및 반응형 축소
  - 메뉴명: 1~2줄 허용, 초과 시 말줄임표 처리
  - 가격: 원 단위 콤마 표기(예: `4,000원`)
  - 옵션 체크박스
    - 각 옵션은 독립적으로 선택 가능
    - 옵션 선택 시 라인 아이템 가격 계산에 반영됨
  - `담기` 버튼 동작
    - 클릭/Enter 시 현재 카드의 메뉴 + 선택된 옵션들을 장바구니에 추가
    - 성공 시: 사용자 피드백(토스트 등) 제공, 카드 옵션 체크 상태는 초기화(기본: 모두 해제)
    - 실패 시(재고 부족 등 서버 응답 기준): 실패 메시지 표시
- 장바구니 패널
  - 항목 추가/병합 규칙
    - 같은 메뉴 + 동일 옵션 조합: 기존 라인 수량 1 증가(라인 소계 금액 갱신)
    - 같은 메뉴 + 다른 옵션 조합: 별도 라인으로 추가
  - 총 금액 계산: 모든 라인 소계를 합산, 0원일 경우 `총 금액 0원` 표시
  - 빈 상태: `장바구니가 비어 있습니다.` 문구 표시, `주문하기` 버튼 비활성화
  - 편집 기능(1차 범위)
    - 수량 증가는 동일 아이템을 다시 `담기` 하는 방식으로만 제공
    - 삭제/감소 기능은 2차 확장 항목으로 둠(라인 삭제, `+`/`-` 수량 조절)
- 하단 `주문하기` 버튼
  - 활성 조건: 장바구니에 1개 이상 항목이 있을 때만 활성화
  - 클릭 시: 주문 생성 API 호출
    - 성공: 장바구니 초기화, 성공 메시지 표시(예: “주문이 접수되었습니다.”)
    - 실패: 실패 사유 기반 에러 메시지 표시
  - 로딩 상태: 주문 요청 중 버튼 비활성화 및 로딩 표시(예: `주문 처리중...`)

#### 4.1.5 프런트엔드 상태 및 데이터 모델(UI 관점)
- 메뉴 데이터(예시)
  - `id`, `name`, `temperature`, `basePrice`, `description`, `imageUrl`, `options[]`
  - `options[]`: `{ id, label, extraPrice }`
- 장바구니 아이템(예시)
  - `menuId`, `menuName`, `options[]`(선택 옵션), `unitPrice`, `quantity`, `lineTotal`
- 페이지 상태
  - `menus`: 메뉴 목록(fetch)
  - `cartItems`: 장바구니 상태(주문 완료 시 초기화)
  - `isSubmittingOrder`: 주문 요청 로딩 플래그
  - `errorMessage` 또는 토스트 상태

#### 4.1.6 예외/에러 처리
- 네트워크 오류
  - 메뉴 로딩 실패: 에러 메시지 + 재시도 버튼 표시
  - 주문 생성 실패: 토스트/모달로 실패 안내
- 유효성 검증
  - 장바구니가 비어 있으면 `주문하기` 버튼 비활성화(접근성 도구 입력에도 방어)
- 서버 에러(재고 부족/가격 불일치 등)
  - 서버 메시지가 있으면 우선 노출, 없으면 공통 메시지 노출

#### 4.1.7 반응형 & UI 가이드
- 브라우저 타겟: 데스크톱 웹(최소 1280px 폭 기준) 우선
- 반응형 규칙
  - 넓은 화면: 메뉴 카드 3개 이상 가로 배치
  - 좁은 화면: 2개/1개로 재배치하며 행 수 증가
  - 장바구니 패널: 하단 전체 폭 유지
- 스타일 가이드(고수준)
  - 카드/패널/버튼의 테두리/라운드/그림자 스타일 일관 유지
  - 주요 액션(`주문하기`) 버튼을 시각적으로 강조
  - 가격/총액은 굵게 표시

#### 4.1.8 추적 지표(선택)
- 기본 이벤트
  - 메뉴별 `담기` 클릭 횟수
  - `주문하기` 클릭 횟수 및 성공/실패 건수
- 파생 지표
  - 주문당 평균 메뉴 수량
  - 옵션 선택 비율(예: 샷 추가 선택 비율)
